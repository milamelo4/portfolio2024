Debugging in the browser

I think this is my favorite topic. A while ago I figure it out that the way I understand the abstract of programming was debugging. I'm a visual learner and I need to see it happening. That being said lets jump right in.

The command debugger can be use to pause the code to be analyzed, however, it only works if Dev tools are open.

Tracing the execution:

  • "Resume": continue the execution.
  • “Step”: run the next command.
  • “Step over”: run the next command, but don't go into a function.
  • “Step into”: Similar to Step, but behaves differently in case of asynchronous function calls.
  • “Step out”: continue the execution till the end of the current function
  • enable/disable all breakpoints.
IMPORTANT

“Step” command ignores async actions, such as setTimeout (scheduled function call), that execute later. The “Step into” goes into their code, waiting for them if necessary.

Codding Style

I'm all about using a clean an easy code and I really enjoyed al the suggested rules...until I saw A semicolon; is mandatory. It might look more organized but I know is not mandatory. Interestingly enough, because I'm dyslexic the less symbols the better, therefore I don't use them. However, style lint can help with that.

Cheat sheet suggestions:

  1. No space between the function name and parentheses between the parentheses and the parameter.
  2. A space between parameters
  3. Curly brace { on the same line, after a space
  4. Spaces around operators
  5. A space between arguments
  6. Lines are not very long
  7. else { without a line break
  8. Spaces around a nested call: { import }
  9. An empty line between logical block
  10. Indentation 2 spaces
  11. A space after for/if/while

Error handling, "try...catch"

I've learned how to use try and catch before, but I definitely need some revision.

  1. First, the code in try {...} is executed.
  2. f there were no errors, then catch (err) is ignored: the execution reaches the end of try and goes on, skipping catch.
  3. If an error occurs, then the try execution is stopped, and control flows to the beginning of catch (err). The err variable (we can use any name for it) will contain an error object with details about what happened.

Steps of execution:

  • Begin
    • Try {code}
  • No Errors
    • -> catch block ignored
  • Error occurred
    • -> Ignore the rest of try
    -> execute catch block

In some ways, a try and catch block in JavaScript can be similar to an if statement with a conditional check. I think the main difference is that after the catch block handles the error, the program continues executing the code after the try-catch block.

In our case, try...catch is placed to catch “incorrect data” errors. But by its nature, catch gets all errors from try.
